# Deixis v4.0.1 — feed layout default + Settings panel design

**Status:** Design approved 2026-06-20
**Author:** Jakub Dudek
**Builds on:** v4.0. Web-only (no hub/shim/shared changes).

---

## 1. Summary

The dashboard currently always uses the Pinterest-style masonry grid (v3). v4.0.1
makes the default a **single-column feed** (Instagram-style) and moves masonry to
an opt-in **Setting**. A new **Settings panel** (opened from a `⚙` in the header)
holds the layout choice and the theme control. The preference persists in
`localStorage`, like the theme.

Entirely within `packages/web` — no protocol, hub, shim, or shared changes.

---

## 2. Layout modes

`Grid` gains a `layout` prop: `"feed" | "masonry"`.

- **`feed` (default):** one **centered column**, `max-width: 600px`, `margin: 0
  auto`, cards stacked vertically at full column width. The masonry "no row-gaps"
  concern doesn't apply (each card is its own row). Motion enter/exit preserved;
  the `onOpenDoc` doc affordance threads through.
- **`masonry`:** the existing v3 implementation unchanged — native CSS
  `grid-template-rows: masonry` where supported, else the round-robin flex-column
  fallback. Already responsive (`min-w-0`, width-safe track).

`Grid` branches on `layout` at the top; the empty-state and `Card` helper are
shared across both.

---

## 3. `useLayout` hook

`packages/web/src/lib/layout.ts` — mirrors the existing `useTheme`:

- `useLayout(): { layout: "feed" | "masonry"; setLayout: (l) => void }`.
- Initial value: `localStorage["deixis-layout"]` if a valid value, else `"feed"`
  (the default).
- On change, persists to `localStorage["deixis-layout"]`.
- No DOM side effects (unlike `useTheme`, which toggles the `.dark` class) — the
  value is just passed to `Grid`.

---

## 4. Settings panel

`packages/web/src/components/SettingsPanel.tsx` — a modal overlay reusing the
v4.0 reading-overlay pattern (fixed backdrop, centered card, `Esc`/backdrop/`✕`
close, inner-click `stopPropagation`).

`SettingsPanel` props: `{ theme, onToggleTheme, layout, onSetLayout, onClose }`
(it is presentational — it does NOT call `useTheme`/`useLayout` itself; see §5 for
why).

Sections:
- **Appearance** — a light/dark control bound to the `theme`/`onToggleTheme`
  props (a labeled toggle/segmented control).
- **Layout** — a two-option control (radio/segmented): **Feed** (default) /
  **Masonry**, bound to `layout`/`onSetLayout`.

Styling: monochrome, `prose`/card conventions already in use; Tailwind v4 generated
utilities (not `[--color-*]`). Light/dark aware.

---

## 5. Header & App wiring

- **Header:** replace the standalone `ThemeToggle` button with a `⚙` **Settings**
  button. Header reads `Deixis … ⚙`.
- **`App`** owns the hooks (so they're always mounted):
  - `const { theme, toggle } = useTheme()` — **must be called at App level**, not
    only inside the conditionally-rendered panel, so the `.dark`-class effect runs
    whenever the app is open, not just when Settings is open.
  - `const { layout, setLayout } = useLayout()`, passed to `<Grid layout={layout}>`.
  - a `settingsOpen` state; the `⚙` button toggles it; renders
    `<SettingsPanel theme={theme} onToggleTheme={toggle} layout={layout}
    onSetLayout={setLayout} onClose={...}>` when open.
- The old `ThemeToggle` component is removed (its `useTheme` call + control are
  superseded by App's call + the panel's Appearance control), to avoid two
  `useTheme` instances both toggling `.dark`.

---

## 6. Package shape

```
packages/web/src/
  lib/layout.ts                  # NEW: useLayout hook
  components/SettingsPanel.tsx   # NEW: presentational ⚙ panel (Appearance + Layout)
  components/Grid.tsx            # + layout prop; feed branch
  components/ThemeToggle.tsx     # REMOVED (App owns useTheme; panel renders the control)
  App.tsx                        # useTheme + useLayout at App level; ⚙ button + SettingsPanel; Grid layout
```
(`menubar`, `hub`, `shim`, `shared`, `cli` unchanged.)

---

## 7. Non-goals

No server-side / cross-device persistence (localStorage only) · no layouts beyond
feed/masonry · no settings beyond theme + layout · no changes outside
`packages/web` · no new test harness for web (manual-verified, consistent with
v1–v4).

---

## 8. Testing

Web is manual-verified (no vitest in `packages/web`, as throughout v1–v4):
- `pnpm --filter @deixis/web build` clean.
- Eyeball: default is the centered feed; `⚙` opens Settings; switching to Masonry
  re-renders the grid; theme control in the panel still toggles light/dark; both
  choices persist across a refresh (localStorage); feed column is centered and
  capped at ~600px; doc affordance + collapse still work in feed mode.
