# Deixis — agent-curated visual canvas for Claude Code

**Status:** Design approved 2026-06-20 (implementation decisions locked)
**Author:** Jakub Dudek
**Scope:** v1 (canvas only). v2 (telemetry) sketched but out of scope.

---

## 1. Summary

Deixis is a persistent browser canvas where each Claude Code session gets its own
view and **the agent chooses what to display there** — a Markdown explanation and
a live, rich progress checklist. All active sessions are shown together in a live
grid.

The name fits: *deixis* is the class of pointing words ("this", "here", "look").
Deixis gives a terminal agent a way to point at things.

### Why it exists (the gap)

Two adjacent categories already exist; Deixis sits in the gap between them:

- **Passive monitors** (claude-view, claude-code-monitor, claude-control)
  *scrape* session state — tokens, cost, status — via hooks/JSONL. The agent does
  not decide what they show.
- **Agent-pushed UI** (MCP Apps / MCP-UI, official Jan 2026) lets a tool return
  interactive HTML, but it renders **inline inside GUI chat clients**. Claude
  Code is a terminal and cannot render HTML inline, so that standard does not help
  terminal sessions.

Deixis is the unoccupied combination: **an agent-*curated*, persistent, per-session
visual surface for the *terminal* Claude Code**, where the model chooses what to
show rather than a dashboard scraping telemetry.

## 2. Architecture

Three runtime pieces plus a browser app. Each has one job.

```
  CC session A ─┐                              ┌─ grid card "A"
                │   (stdio MCP, per session)   │
  CC session B ─┼──> MCP shim ──HTTP POST──>  HUB ──SSE──> dashboard (React)
                │                            (:3000)        live grid of all
  CC session C ─┘                                           sessions
```

### 2.1 Hub
- Long-running local server on `localhost:3000`, started as an always-on service.
- Holds all state **in memory**: `Map<sessionId, SessionState>` (no persistence).
- Serves the built React dashboard (static assets) and the SSE stream.
- Pushes state changes to all connected browsers over **Server-Sent Events** —
  one-way (server → browser), which is all a display needs.

### 2.2 MCP shim
- A tiny stdio MCP server. Claude Code launches **one per session
  automatically**, which gives each session a stable identity for free.
- Holds no state, no UI. Each tool call performs an HTTP POST to the hub using a
  **bespoke JSON protocol** (a small message shape invented for this project; only
  the shim and hub understand it — no external standard needed because no stranger
  talks to the hub).

### 2.3 Dashboard (React)
- A live **grid** of cards, one per active session, all updating simultaneously.
- Each card renders that session's rich progress (top) and markdown (below).
- Transitions/enter-exit animated with `motion`.
- Light + dark themes, following the OS with a manual toggle.

### 2.4 Lifecycle (always-on service)
- A macOS **launchd** user agent auto-starts the hub at login and restarts it on
  crash. Installed/removed by the `deixis` CLI (Section 8). Logs to a file under
  the user's Library/Logs.

## 3. Tools (the entire v1 surface)

| Tool | Signature | Effect |
|---|---|---|
| `render_markdown` | `(markdown: string)` | Replaces the session's markdown pane. GFM, so tables/code/images come free. |
| `progress_set` | `(steps: Step[])` | Defines/replaces the checklist (supports nested substeps). |
| `progress_update` | `(stepId: string, status: Status, note?: string)` | Updates one step without resending the list; hub stamps timing. |

### 3.1 Rich progress model

```
Status = "pending" | "active" | "done" | "failed" | "blocked"

Step = {
  id: string,
  name: string,
  status: Status,
  note?: string,          // short per-step annotation
  substeps?: Step[],      // one level of nesting (v1)
  startedAt?: number,     // stamped by hub on -> active
  endedAt?: number,       // stamped by hub on -> done | failed
}
```

- `progress_set` accepts/assigns step `id`s so `progress_update` can target one.
- The hub computes timing on status transitions (active sets `startedAt`;
  done/failed set `endedAt`); the agent never sends timestamps.
- Card header shows a glanceable count and a thin progress bar (done / total).
- Markdown and progress are independent panes; either may be empty.

## 4. Identity & lifecycle

- **Label:** auto-derived from the session's working-directory folder name;
  duplicate names disambiguated with a numeric suffix (`deixis`, `deixis-2`).
- **Session id:** generated by the shim at startup; stable for the session's life.
- **State:** in-memory only. A hub restart clears all cards; sessions re-render on
  their next tool call.
- **Offline:** when a shim process exits, it best-effort notifies the hub; the
  card greys out, then is removed after a short grace period to keep the grid
  clean.

---

## 5. Bespoke protocol (shim → hub)

HTTP POST, JSON bodies. Indicative shapes (final endpoint names settled in plan):

```
POST /session/:sessionId/register          { label }
POST /session/:sessionId/markdown          { markdown }
POST /session/:sessionId/progress          { steps: Step[] }
POST /session/:sessionId/progress/update   { stepId, status, note? }
POST /session/:sessionId/disconnect        {}
GET  /events                               (SSE stream consumed by the dashboard)
```

Shared TypeScript types for these shapes live in the `shared` package and are
imported by both `hub` and `shim` so the protocol cannot drift.

---

## 6. Error handling

- If the hub is not running, the shim's POST fails and the tool returns a friendly
  message (e.g. "Deixis canvas not running on :3000"). **Non-fatal** — the session
  continues; nothing it relies on breaks.
- The launchd service makes "hub not running" rare in practice.

## 7. Stack & design

### 7.1 Tech stack
- **Language:** TypeScript everywhere, with a build step.
- **Monorepo:** pnpm workspaces (Section 8.1).
- **Hub:** Node LTS HTTP server + SSE; serves the built dashboard.
- **Shim:** Node + MCP SDK over stdio.
- **Dashboard:** React + Vite + Tailwind v4 (`@theme` tokens) + shadcn
  (`base-nova`) + `motion` for transitions. Icons: Lucide.

### 7.2 Design language (matches pollar/apps/frontend)

A calm monochrome base so the chrome never fights the agent's content.

- **Color — monochrome, OKLCH.** Light: bg `oklch(1 0 0)`, fg
  `oklch(0.145 0 0)`, muted-fg `oklch(0.556 0 0)`, border `oklch(0.922 0 0)`.
  Dark: bg `oklch(0.145 0 0)`, fg `oklch(0.985 0 0)`, card `oklch(0.205 0 0)`,
  borders via `oklch(1 0 0 / 10%)`.
- **Semantic status colors** (the one deliberate exception to monochrome), tuned
  per theme in OKLCH — indicative: `done` green, `active` amber, `failed` red,
  `blocked` muted/gray, `pending` low-contrast gray. Final values tuned in
  implementation for contrast in both themes.
- **Type — two weights only.** 400 body, 500 all emphasis/headings; never
  heavier. Small scale: ~11px uppercase eyebrows (`letter-spacing: 0.05em`),
  13–15px body, ~20px titles. Tight leading on headings, relaxed on body.
- **Flat.** No shadows except, if needed, a single subtle one on a floating card.
  Borders + opacity do the work.
- **Radius.** Base `0.625rem` (10px) with `calc()` multipliers (sm 6px → 2xl 18px).
- **Motion.** 300ms `cubic-bezier(0.4, 0, 0.2, 1)`; smooth, not bouncy.
- **Theme:** light + dark, auto-following OS, manual toggle, preference persisted
  in `localStorage`.

### 7.3 Fonts (real files, sourced locally)
- **Sans (UI + headings): Helvetica Now Display.** Convert two weights from the
  user's local `~/Downloads/Helvetica Now/` `.otf` to `woff2` and bundle:
  - `HelveticaNowDisplay.otf` → 400
  - `HelveticaNowDisplayMedium.otf` → 500
- **Mono (markdown code): PPSupplyMono.** Reuse
  `pollar/apps/frontend/src/app/fonts/PPSupplyMono-Regular.woff2` (400).
- **Loading:** `@font-face`, `font-display: swap`, fallback to a metrically close
  system grotesque to avoid layout shift.
- **License:** Helvetica Now and PPSupplyMono are proprietary; these are the
  user's own licensed files, bundled locally for a personal tool, not
  redistributed. (The published package must NOT ship the fonts; the installer
  converts them from the user's local copy — see 8.2.)

## 8. Distribution

### 8.1 Monorepo layout (pnpm workspaces)
```
deixis/
  pnpm-workspace.yaml
  package.json
  packages/
    shared/   # protocol & domain types (Step, Status, message shapes)
    hub/      # HTTP + SSE server, in-memory store, serves dashboard
    shim/     # MCP stdio server (the 3 tools) -> POSTs to hub
    web/      # React + Vite + Tailwind v4 + shadcn dashboard
    cli/      # `deixis` command: init / install / uninstall
  docs/
  assets/fonts/   # generated woff2 (gitignored; produced by installer)
```

### 8.2 `deixis` CLI (publishable package + installer)
`deixis init` performs first-run setup:
1. Registers the MCP shim in the user's Claude Code config (so every session
   gets the tools).
2. Installs and loads the launchd service for the hub.
3. Locates the user's Helvetica Now `.otf` files, converts the two weights to
   `woff2`, and places them in `assets/fonts/` (fonts are not shipped in the
   package for licensing reasons).
Plus `deixis uninstall` (unload service, remove MCP entry) and `deixis status`.

## 9. Testing

- **Unit:** hub endpoints (POST → in-memory state assertions), SSE broadcast,
  rich-progress transitions/timing, and the shim's per-tool POST shapes.
- **Manual:** verify rendering and live grid updates by eye in the browser.
- No end-to-end browser automation in v1 (deliberate; revisit if the UI grows).

---

## 10. Explicit v1 non-goals (all reversible)

No persistent storage (in-memory only) · no WebSocket (SSE is sufficient) · no
auth · no Claude Code hooks · no coordination / notes / mail · no arbitrary HTML
or JS execution · no telemetry · no e2e test harness.

---

## 11. v2 — designed-for, not built

A **telemetry pipe**: Claude Code hooks (`SessionStart`, `PreToolUse`, `Stop`, …)
plus session JSONL feed automatic status (working / idle / waiting, tokens, cost)
into the **same hub and the same dashboard** — a second data pipe, not a second
project. Sequenced as v2 because it **requires hooks**, the one ingredient v1
deliberately avoids to keep the canvas safe and simple.
